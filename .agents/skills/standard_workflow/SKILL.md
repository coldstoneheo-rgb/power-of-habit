# 표준 워크플로우 지침

이 스킬은 프로젝트 내 모든 작업에서 사용자의 개별 지시가 없더라도 표준 워크플로우에 따라 작업을 자동 설계, 진행, 검증하도록 통제하는 지침서입니다.

## 표준 워크플로우 12단계

1. **현 상태 동기화**
   - 현재 브랜치, `main`/원격 동기화 상태, 미커밋 변경, untracked 파일을 먼저 확인한다.
   - 이전 PR이 머지됐다면 로컬 `main`으로 전환 후 `git pull --ff-only`.
   - `scratch/`, 테스트 산출물, 로컬 CLI 설정 파일 등은 커밋 대상에서 제외한다.

2. **다음 작업 선정**
   - 억지로 일을 만들지 않는다.
   - 현재 목표 기준으로 “지금 가장 큰 리스크를 줄이는 작업”을 고른다.
   - 베타/출시 직전이면 새 기능보다 로그인, 권한, 핵심 화면, 관측 가능성, 사용자 안내를 우선한다.
   - 이미 충분한 품질이면 “이제 시작하자”는 판단도 작업 결과로 인정한다.

3. **선정 이유 제시**
   - 왜 지금 이 작업이 필요한지 설명한다.
   - 사용자 가치, 리스크, 일정, 데이터 신뢰성, UX 영향 기준으로 판단한다.
   - 다른 후보가 있다면 왜 후순위인지 짧게 밝힌다.

4. **구체 계획 설계**
   - 작업 범위, 비범위, 변경 예상 파일, 데이터 흐름, UI 영향, 검증 방법을 명확히 적는다.
   - 특히 DB, 권한, 인증, 결제, 개인정보와 관련된 작업은 강하게 주의 표시한다.
   - 계획에는 다음 항목을 포함한다:
     - 현재 상태
     - 선정 작업
     - 선정 이유
     - 작업 계획 A/B/C...
     - 검증 계획
     - 위험 & 완화
     - 필요 스킬
     - 작업 배정

5. **사용자 승인**
   - 원칙적으로 승인 전에는 코드를 수정하지 않는다.
   - “좋아”, “진행해”, “시작하자”, “구현 들어가자” 같은 명시 승인을 받은 뒤 구현한다.
   - 사용자가 자율 구현 모드를 명시하면 승인 단계를 생략할 수 있지만, 그 경우에도 계획은 먼저 세운다.

6. **새 작업 브랜치 생성**
   - 최신 `main`에서 새 브랜치를 만든다.
   - 오래된 브랜치를 재사용하지 않는다.
   - 브랜치명은 작업 목적이 드러나게 짧고 구체적으로 짓는다.

7. **구현**
   - 승인된 계획 안에서만 작업한다.
   - 범위가 커지거나 위험도가 올라가면 멈추고 다시 보고한다.
   - 사용자/다른 에이전트가 만든 기존 변경을 되돌리지 않는다.
   - 관련 없는 리팩터링, 포맷 churn, 파일 정리는 피한다.
   - 커밋에는 의도한 파일만 stage한다.

8. **구현 후 검증**
   - 먼저 `git status --short --branch`, `git diff --stat`, 핵심 diff를 읽는다.
   - 변경 성격에 맞는 타깃 테스트를 먼저 실행한다.
   - 코드/UI/라우트 변경이면 보통 아래를 고려한다:
     - `npm.cmd run lint`
     - `npm.cmd run build`
     - 관련 Playwright/E2E 또는 단위 테스트
     - `git diff --check`
   - 단순히 “통과했다”가 아니라 왜 안전한지 설명한다.
   - 결과를 `PASS / MINOR / BLOCKER`로 분류한다.

9. **커밋, 푸시, PR 생성**
   - 의도한 파일만 stage한다.
   - 커밋 메시지는 작업 의도를 짧게 표현한다.
   - 원격 브랜치에 push한다.
   - PR 본문에는 `Summary`, `Verification`, `Notes`를 쓴다.
   - 검증하지 않은 내용을 PR 본문에 적지 않는다.

10. **PR 리뷰 확인**
    - Codex/Gemini/Vercel/Supabase 리뷰와 체크를 확인한다.
    - 리뷰봇의 제안을 무조건 따르지 않는다.
    - 프로젝트 맥락상 타당한지, 정상 코드를 망가뜨리지 않는지 먼저 판단한다.
    - 타당한 지적만 좁게 수정하고 재검증한다.
    - 리뷰 댓글에 대응 내용을 남기고 resolved 처리한다.

11. **머지 루틴**
    - 리뷰 댓글을 resolved 처리 했으면 머지 후 브랜치를 삭제한다.
    - 머지 후 로컬은 `main`으로 전환, `git pull --ff-only`, 로컬 작업 브랜치 삭제.
    - 삭제하면 안 되는 브랜치라면 사용자에게 이유를 말한다.

12. **다음 작업으로 회귀**
    - 다시 상태 동기화부터 시작한다.
    - 이전 작업에서 발견한 후속 과제는 즉시 구현하지 말고 우선순위에 넣는다.

## 에이전트/모델 역할 분담

- **고추론 모델**: 작업 선정, 계획 설계, 위험 판단, PR 리뷰 타당성 검토, 검증 결과 해석, DB/권한/보안/데이터 신뢰성 판단
- **구현 에이전트**: 승인된 계획에 따른 코드 수정, 단순 문서 작업, 테스트 보강, 커밋/푸시/PR 생성
- **단순 점검 에이전트**: git 상태 확인, 로그/테스트 결과 수집, PR 상태/체크 상태 확인, 반복적인 smoke test 실행

## 핵심 수칙 및 가이드라인

- **사용자 가치 우선**: “사용자가 실제로 얻는 가치가 좋아졌는가?”, “데이터 신뢰성이 높아졌는가?”를 항상 질문합니다.
- **DB 작업 엄격 관리**: 마이그레이션 idempotency 확인, schema mismatch 방지, 구조 변경 시 계획/검증/롤백 전략 상시 수립.
- **PR 소형화**: 1 PR = 1 의도 원칙을 준수하고 리팩터링 및 UI를 과도하게 혼재시키지 않습니다.
